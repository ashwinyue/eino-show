# 对话评估计划

> 本计划用于评估已实现的对话功能，暂不考虑 RAG 相关能力。

## 一、评估目标

验证对话系统的核心能力，包括：
- 基础对话质量
- Agent 推理能力
- 多 Agent 协作
- 会话管理
- 流式响应

---

## 二、评估维度

### 2.1 Chat Agent 基础对话

| 评估项 | 描述 | 评估方法 |
|--------|------|----------|
| 响应质量 | 回答的准确性、完整性、流畅性 | 人工评分 (1-5 分) |
| 上下文理解 | 多轮对话中的上下文保持能力 | 对话连贯性测试 |
| 指令遵循 | 对用户指令的执行准确度 | 指令测试集 |
| 响应时延 | 首 Token 时间 (TTFT) 和总时间 | 自动化性能测试 |

### 2.2 ReAct Agent 推理能力

| 评估项 | 描述 | 评估方法 |
|--------|------|----------|
| 任务分解 | 复杂任务的拆解能力 | 多步骤任务测试 |
| 工具选择 | 正确选择工具的能力 | 工具选择准确率 |
| 工具调用 | 工具参数构建正确性 | 参数正确率 |
| 迭代推理 | 多轮思考-行动-观察循环 | 推理深度测试 |
| 错误恢复 | 工具调用失败后的处理 | 异常场景测试 |

### 2.3 Supervisor 多 Agent 协作

| 评估项 | 描述 | 评估方法 |
|--------|------|----------|
| Agent 路由 | 正确分派任务到子 Agent | 路由准确率 |
| 协作效率 | 多 Agent 协作的总耗时 | 性能基准测试 |
| 结果整合 | 多个 Agent 结果的汇总质量 | 输出质量评分 |

### 2.4 Workflow 工作流

| 评估项 | 描述 | 评估方法 |
|--------|------|----------|
| Sequential 顺序执行 | 顺序执行的正确性 | 流程验证 |
| Loop 循环执行 | 循环条件判断和终止 | 边界条件测试 |
| Parallel 并行执行 | 并行任务的正确合并 | 结果一致性检查 |

### 2.5 会话管理

| 评估项 | 描述 | 评估方法 |
|--------|------|----------|
| 历史保存 | 消息正确持久化 | 数据一致性验证 |
| 上下文压缩 | 长对话的上下文管理 | 长对话测试 |
| 标题生成 | 自动生成有意义的标题 | 标题相关性评估 |

### 2.6 流式响应

| 评估项 | 描述 | 评估方法 |
|--------|------|----------|
| SSE 稳定性 | 事件流的连续性 | 长时间连接测试 |
| 事件完整性 | 所有事件类型正确发送 | 事件覆盖率 |
| 断线重连 | 网络异常后的恢复 | 异常模拟测试 |

## 三、测试用例设计

### 3.1 Chat Agent 测试用例

```
用例 CH-001: 简单问答
- 输入: "你好，介绍一下你自己"
- 预期: 返回自我介绍，语言流畅

用例 CH-002: 多轮对话上下文
- 第1轮: "我叫小明"
- 第2轮: "我刚才说我叫什么名字？"
- 预期: 正确回忆 "小明"

用例 CH-003: 长对话 (10+ 轮)
- 连续进行10轮以上对话
- 预期: 上下文压缩后仍能保持核心信息

用例 CH-004: 指令遵循
- 输入: "用三句话描述地球"
- 预期: 严格输出三句话
```

### 3.2 ReAct Agent 测试用例

```
用例 RA-001: 单工具调用
- 输入: "搜索今天的新闻"
- 预期: 正确调用 web_search 工具

用例 RA-002: 多工具链式调用
- 输入: "先搜索 XX 的信息，然后用 grep_chunks 在知识库中查找相关内容"
- 预期: 依次调用 web_search, grep_chunks

用例 RA-003: 工具调用失败恢复
- 模拟: web_search 返回错误
- 预期: Agent 尝试替代方案或告知用户

用例 RA-004: 无需工具的问题
- 输入: "1+1等于几"
- 预期: 直接回答，不调用工具
```

### 3.3 Supervisor 测试用例

```
用例 SV-001: 任务分派
- 配置: 两个子 Agent (快速问答、智能推理)
- 输入: "帮我分析这个复杂问题..."
- 预期: 分派给智能推理 Agent

用例 SV-002: 子 Agent 协作
- 输入: 需要多个 Agent 协作的任务
- 预期: Supervisor 正确协调，结果完整
```

### 3.4 Workflow 测试用例

```
用例 WF-001: 顺序工作流
- 配置: Agent A → Agent B → Agent C
- 预期: 按顺序执行，前一个的输出作为后一个的输入

用例 WF-002: 循环工作流
- 配置: 最大迭代 3 次
- 预期: 正确执行 3 次后终止

用例 WF-003: 并行工作流
- 配置: 3 个 Agent 并行执行
- 预期: 结果正确合并
```

## 六、评估流程

### 阶段 1: 单元测试 (自动化)

```bash
# 运行所有单元测试
make test

# 运行特定模块测试
go test -v ./internal/pkg/agent/...
```

### 阶段 2: 集成测试 (自动化)

```bash
# 启动测试环境
docker-compose -f docker-compose.test.yml up -d

# 运行集成测试
go test -v ./tests/integration/...
```

### 阶段 3: 功能验收测试 (手动)

1. 按测试用例逐一验证
2. 记录每个用例的通过/失败状态
3. 对失败用例进行问题定位

### 阶段 4: 性能压测

```bash
# 使用 wrk 进行压测
wrk -t12 -c100 -d30s -s scripts/chat_bench.lua http://localhost:8080/api/v1/
```

### 阶段 5: 评估报告

生成评估报告，包括：
- 测试覆盖率
- 各项指标达成情况
- 发现的问题清单
- 改进建议

## 十、风险与注意事项

1. **模型依赖** - 测试结果受 LLM 模型版本影响，建议固定模型版本
2. **环境一致性** - 确保测试环境与生产环境配置一致
3. **成本控制** - 大规模测试会产生 API 调用费用，建议设置预算上限
4. **幂等性** - 部分测试可能因模型随机性导致结果不一致
