# eino-show 项目规约

## 开发流程

### 首次启动

```bash
# 1. 创建环境配置
cp .env.example .env

# 2. 启动基础设施（PostgreSQL + Redis）
make dev-start

# 3. 等待数据库初始化完成（约30秒）
# 首次启动会自动从 WeKnora 迁移数据

# 4. 在新终端启动后端应用
make dev-app
```

### 日常开发

```bash
# 启动开发环境
make dev-start    # 启动 PostgreSQL + Redis

# 启动后端应用（热重载）
make dev-app      # 使用 Air 实现 go 代码修改后自动重启

# 查看服务状态
make dev-status

# 查看容器日志
make dev-logs

# 停止服务
make dev-stop
```

### 代码修改后

```bash
# 每次修改代码后自动运行 build 检查
# （Air 会自动检测并重新编译）

# 手动编译检查（统一输出到 bin/ 目录）
make build              # 等同于: go build -o bin/es-apiserver ./cmd/mb-apiserver/main.go
```

### Proto 文件修改

```bash
# 修改 .proto 文件后，重新生成代码
make gen.protoc

# 验证编译
go build ./pkg/api/apiserver/...
```

- **单个微服务模块** 放在 `internal/{module}/`，如 `internal/apiserver/`
- **跨服务的业务工具包** 放在 `internal/pkg/{package}/`，如 `internal/pkg/agent/`
- **通用工具函数** 放在 `pkg/{package}/`，如 `pkg/errors/`、`pkg/utils/`

### GORM 标签规范

生成器自动处理以下标签：
- 主键: `primaryKey`
- 类型: `type:varchar(255)`
- 非空: `not null`
- 索引: `index:idx_name` 或 `uniqueIndex:idx_name`
- 默认值: `default:current_timestamp`
- 可空字段: 自动使用指针类型
- JSONB/JSON: 自动映射为自定义类型

- ❌ 手动创建或编辑 `*.gen.go` 文件
- ❌ 在 `internal/apiserver/model/` 目录下添加非生成文件
- ❌ 直接修改生成器输出的代码

### 接口定义模板
```go
// SessionStore 定义了 session 模块在 store 层所实现的方法.
type SessionStore interface {
    Create(ctx context.Context, obj *model.SessionM) error
    Update(ctx context.Context, obj *model.SessionM) error
    Delete(ctx context.Context, opts *where.Options) error
    Get(ctx context.Context, opts *where.Options) (*model.SessionM, error)
    List(ctx context.Context, opts *where.Options) (int64, []*model.SessionM, error)

    SessionExpansion  // 扩展方法
}

// SessionExpansion 定义了会话操作的附加方法.
type SessionExpansion interface {
    GetByTenantID(ctx context.Context, tenantID uint64) ([]*model.SessionM, error)
}
```

### 实现模板
```go
type sessionStore struct {
    *genericstore.Store[model.SessionM]
}

var _ SessionStore = (*sessionStore)(nil)

func newSessionStore(store *datastore) *sessionStore {
    return &sessionStore{
        Store: genericstore.NewStore[model.SessionM](store, NewLogger()),
    }
}
```

### IStore 扩展
在 `store.go` 的 `IStore` 接口中添加：

```go
    User() UserStore

    // ...
}
```

在 `datastore` 结构体中添加：

### 提示词语言规范（对齐 WeKnora）

**核心原则**: 所有传给 LLM 的内容（Agent Name、Tool Name、Description、System Prompt）**必须使用英文**。

#### 为什么必须是英文？

1. **模型效果**: LLM 在英文 prompt 下的理解和推理能力更强
2. **API 兼容性**: 与 WeKnora 保持一致，确保前端无需修改
3. **最佳实践**: 业界标准做法

#### 双层命名规范

| 用途 | 语言 | 示例 | 说明 |
|------|------|------|------|
| **传给 LLM** | 英文 | `knowledge_search` | Tool.Info() 返回的 Name、Desc |
| **前端显示** | 中文 | `语义搜索` | API 返回的 label、description |

#### 代码示例

```go
// internal/pkg/agent/tool/knowledge_search.go

func (t *KnowledgeSearch) Info(_ context.Context) (*schema.ToolInfo, error) {
    return &schema.ToolInfo{
        Name: "knowledge_search",  // 英文 - 传给 LLM
        Desc: `Semantic/vector search tool for retrieving knowledge by meaning...

This tool searches by MEANING rather than exact text...`,  // 英文 - 传给 LLM
        ParamsOneOf: schema.NewParamsOneOfByJSONSchema(
            jsonschema.Reflect(knowledgeSearchArgs{}),
        ),
    }, nil
}

type knowledgeSearchArgs struct {
    Query string `json:"query" jsonschema:"description=REQUIRED: A semantic question...,required"`  // 英文
}

// 前端显示用的中文定义（在别处，如 builtin.go）
var AvailableTools = []AvailableTool{
    {
        Name: "knowledge_search",       // 英文 ID
        Label: "语义搜索",              // 中文 - 前端显示
        Description: "理解问题并查找语义相关内容",  // 中文 - 前端显示
    },
}
```

#### 规范对照表

| 代码位置 | 内容类型 | 语言要求 | 作用对象 |
|----------|----------|----------|----------|
| `Tool.Info().Name` | Tool 名称 | 英文 | LLM |
| `Tool.Info().Desc` | Tool 描述 | 英文 | LLM |
| `jsonschema:"description=..."` | 参数描述 | 英文 | LLM |
